## Recommended Stack

### Core Technologies

| Technology | Version | Purpose | Why Recommended |
|------------|---------|---------|-----------------|
| Svelte 5 | ≥5.46.4 | UI framework | Runes API is canonical Svelte 5 reactivity; compiles to minimal JS with no runtime overhead — ideal for a single-screen SPA that must stay lean in-browser |
| Vite 8 | ^8.0.0 | Build tool / dev server | Required by `@sveltejs/vite-plugin-svelte` v7+; native ESM, fastest HMR, excellent tree-shaking; `vite build` → static output hostable anywhere |
| TypeScript 5 | ^5.0.0 | Type safety | Strict mode enforced by project spec; Svelte 5 runes are fully TypeScript-aware with no `any` escape hatches required |
| `@contextvm/sdk` | 0.6.2 | MCP-over-Nostr transport | Pinned by project spec; `NostrServerTransport` abstracts relay pooling, NIP-04/44 encryption, and MCP framing — building from raw `nostr-tools` would duplicate all of this |
| Web Crypto API | built-in | nsec key encryption | Browser-native AES-256-GCM + PBKDF2; no library needed; `crypto.subtle` is available in all modern browsers and is the right tool for passphrase-based key wrapping |

### Key encryption pattern (Web Crypto API)

```typescript
// derive AES-256-GCM key from passphrase via PBKDF2
async function deriveKey(passphrase: string, salt: Uint8Array): Promise<CryptoKey> {
  const enc = new TextEncoder()
  const base = await crypto.subtle.importKey('raw', enc.encode(passphrase), 'PBKDF2', false, ['deriveKey'])
  return crypto.subtle.deriveKey(
    { name: 'PBKDF2', salt, iterations: 100_000, hash: 'SHA-256' },
    base,
    { name: 'AES-GCM', length: 256 },
    false,
    ['encrypt', 'decrypt']
  )
}
// Store: base64(salt[16] + iv[12] + ciphertext) in localStorage
// Destroy: nsecBytes.fill(0) then localStorage.removeItem(key)
```

## Alternatives Considered

| Recommended | Alternative | When to Use Alternative |
|-------------|-------------|-------------------------|
| Plain Vite + Svelte | SvelteKit | Only if you need SSR, file-based routing, or adapters — this project is a static single-screen SPA with no routing |
| Svelte 5 runes (`$state`, `$derived`, `$effect`) | Svelte stores (`writable`, `derived`) | Never in this project — stores are Svelte 4 legacy; mixing runes and stores in Svelte 5 causes type confusion |
| `@tailwindcss/vite` plugin | PostCSS + `tailwindcss` v4 | Only in monorepos where all apps share a PostCSS config; for a standalone Vite project the Vite plugin is simpler |
| Web Crypto API (`crypto.subtle`) | `tweetnacl` / `libsodium` | Use a library if you need NaCl primitives not in Web Crypto (e.g., XSalsa20-Poly1305); for AES-GCM + PBKDF2 the native API is sufficient and adds zero bundle weight |
| Vitest (Node mode) for unit tests | `@vitest/browser` + `vitest-browser-svelte` | Browser mode is useful for component rendering tests; for pure functions (state machines, key helpers) Node mode is faster and simpler — component rendering is covered by Playwright e2e |
| `@playwright/test` for e2e | Cypress | Playwright is faster in headless CI, has first-class TypeScript support, and its fixture system cleanly supports the mock relay setup this project needs |
| `nsyte` via Deno | `nsyte` via npx | The tool is Deno-native; running it via Node npx wrappers adds shim complexity and may break on version mismatches |

## What NOT to Use

| Avoid | Why | Use Instead |
|-------|-----|-------------|
| SvelteKit | SSR/adapter overhead; `vite build` static output is the deploy target | Plain Vite + `@sveltejs/vite-plugin-svelte` |
| Svelte stores (`writable`, `readable`, `derived`) | Svelte 5 legacy API; runes replace them; mixing causes subtle reactivity bugs and TypeScript friction | `$state()`, `$derived()`, `$effect()` runes |
| `tailwind.config.js` | v4 dropped JS config; attempting to use it causes silent no-ops | CSS-first config via `@import "tailwindcss"` and CSS variables |
| Raw `nostr-tools` WebSocket relay management | `@contextvm/sdk` already abstracts relay pooling, reconnect, and NIP encryption — duplicating it is wasted work and risks protocol divergence | `NostrServerTransport` from `@contextvm/sdk/transport` |
| `@modelcontextprotocol/sdk` directly | The project uses `@contextvm/sdk` which wraps MCP SDK with Nostr transport; importing both creates version conflicts | `@contextvm/sdk` only |
| `jsdom` for Playwright tests | jsdom is a simulated DOM; Playwright provides a real browser; use jsdom only in Vitest unit tests where the DOM is incidental | Real Chromium via `@playwright/test` |
| `localStorage` for raw nsec | Storing unencrypted private key in localStorage exposes it to XSS; any injected script on the page can read it | AES-256-GCM encrypt with PBKDF2-derived key before writing; or keep key in memory only |
| `console.log` for key material during dev | Keys logged to console persist in DevTools history and may be captured by browser extensions | Never log key bytes; use hex prefix (first 8 chars) for debugging at most |

## `@contextvm/sdk` Import Strategy

The SDK ships subpath exports for tree-shaking. Import only what runs in the browser:

```typescript
// CORRECT — tree-shakes server-side gateway/proxy code
import { NostrServerTransport } from '@contextvm/sdk/transport'
import { createLogger } from '@contextvm/sdk/core'

// AVOID — pulls in all exports including Node-only gateway transports
import { NostrServerTransport } from '@contextvm/sdk'
```

Vite will tree-shake unused exports if the package is marked `"sideEffects": false` in its `package.json`, but using the subpath import is safer and more explicit.
